# コントリビューションガイド

「放課後ダッシュ！」への改善提案や開発参加を歓迎します。この文書では、変更を安全かつ確認しやすく進めるための基本ルールを説明します。

## 行動の基本

- 相手を尊重し、提案の背景と目的を丁寧に共有してください。
- 個人への批判ではなく、ゲーム体験、仕様、コード、検証結果を議論してください。
- パスワード、認証コード、個人情報、秘密鍵をIssueやPull Requestへ投稿しないでください。

## 作業を始める前に

1. [Issue一覧](https://github.com/simosimo1333-png/games-5p-gpt/issues)で同じ内容がないか確認します。
2. 小さな修正を除き、先にIssueで目的と完了条件を確認します。
3. 対応するIssueを1つ選び、1つのPull Requestを1つの目的に絞ります。
4. ゲーム仕様を変える場合は `docs/game-design.md`、技術方針を変える場合は `docs/technical-design.md` も更新します。

## ブランチとコミット

- `main` へ直接変更せず、作業用ブランチを作ります。
- ブランチ名は内容が分かる短い英語にします。例: `feature/touch-controls`、`fix/stuck-input`
- コミットメッセージには変更内容を簡潔に書きます。例: `fix: clear touch input on cancel`
- 関係のない変更を同じコミットやPull Requestへ混ぜないでください。

## Pull Requestに書くこと

- 何を変更したか
- なぜ変更したか
- 利用者や開発者への影響
- 確認した端末、ブラウザ、テスト
- 関連Issue（例: `Closes #123`）
- 見た目を変更した場合は、可能であれば変更前後の画像

## 確認項目

開発ツールが整備されるまでは、変更内容に応じて次を手動確認します。

- ローカルWebサーバーからゲームを開ける
- ブラウザの開発者画面に新しいエラーが出ない
- キーボードで左右移動とジャンプができる
- タッチ操作で左右移動とジャンプができる
- 落下後に復帰できる
- ゴール後にタイムが表示され、再挑戦できる
- READMEやドキュメントのリンクが正しい

自動テストとCIの導入後は、リポジトリで定めるすべてのチェックを通してください。

## レビューと取り込み

- まずDraft Pull Requestとして作成し、作業中であることを示します。
- 完了条件と確認結果が揃ったら、レビュー可能な状態へ変更します。
- 指摘へ対応した場合は、どのように対応したか返信します。
- 必須チェックとレビューが完了してから `main` へ取り込みます。
