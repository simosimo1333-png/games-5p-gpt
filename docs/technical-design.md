# 技術設計書

## 1. 設計目標

- 既存の操作確認版を壊さず、段階的に保守可能な構成へ移行する。
- 最大6人のリアルタイム協力プレイで、状態の一貫性と再接続性を確保する。
- モバイルブラウザで安定した描画、入力、通信を実現する。
- クライアント、サーバー、共有プロトコルを独立してテスト・配布できるようにする。

## 2. 推奨技術スタック

| 領域 | 推奨 | 理由 |
| --- | --- | --- |
| 言語 | TypeScript | クライアント・サーバー・通信型を共有できる |
| クライアント | Phaser 3 + Vite | 現行資産を活かしつつ高速な開発環境を得る |
| サーバー | Node.js + WebSocketライブラリ | 小さく始め、リアルタイム同期を実装しやすい |
| パッケージ管理 | npm workspaces | 初期規模で導入負荷が低いモノレポ構成 |
| 単体テスト | Vitest | Vite/TypeScriptとの親和性が高い |
| E2E | Playwright | モバイル表示と複数ブラウザ接続を自動化できる |
| 静的品質 | ESLint + Prettier + TypeScript strict | レビュー前に機械的問題を検出する |
| CI | GitHub Actions | PR単位で型、Lint、テスト、ビルドを検証する |

ライブラリの最終選定は、小規模な技術検証で測定してからADRまたはIssueへ記録する。

## 3. 全体構成

```text
Mobile Browser
  └─ Phaser Client
       ├─ Input sampling
       ├─ Local prediction / interpolation
       └─ WebSocket protocol
              ↕
       Authoritative Game Server
       ├─ Room lifecycle
       ├─ Player/session mapping
       ├─ Fixed-tick simulation
       ├─ Validation and snapshots
       └─ Reconnection handling
```

初期は単一リージョン・単一ゲームサーバープロセスでよい。負荷測定後に、ルーム割り当てと水平分割を検討する。

## 4. クライアント設計

### シーン

- `BootScene`: 最小設定とアセット読込
- `LobbyScene`: ルーム作成・参加、接続状態、準備確認
- `GameScene`: ステージ表示とゲーム進行
- `ResultScene`: 結果、再挑戦、ロビー復帰

### 責務分離

- `InputController`: キーボード、タッチ、キャンセル、端末回転を共通入力へ変換
- `PlayerController`: ローカル操作と表示。通信やUIへ直接依存しない
- `NetworkClient`: 接続、再接続、メッセージ変換、バージョン交渉
- `GameStateStore`: サーバースナップショットと補間用状態を保持
- `StageLoader`: データ駆動で地形、チェックポイント、ギミックを生成
- `Hud`: タイマー、参加者、接続状態、結果表示

## 5. サーバー設計

サーバーをゲーム状態の正本とする。クライアントは入力意図を送り、位置やクリアを最終決定しない。

- ルームID、参加上限、フェーズ、プレイヤー一覧を管理する。
- 固定tickで入力を適用し、検証済み状態を更新する。
- 速度、ジャンプ間隔、能力クールダウンなどを検証する。
- 定期スナップショットと重要イベントを配信する。
- 切断セッションを一定時間保持し、再接続トークンで復帰させる。

## 6. 通信モデル

### メッセージ例

- Client → Server: `join_room`, `ready`, `input`, `use_ability`, `retry_vote`
- Server → Client: `room_state`, `game_started`, `snapshot`, `game_event`, `game_finished`, `error`

全メッセージにプロトコルバージョンを持たせ、共有パッケージのスキーマでランタイム検証する。

### 同期方針

- クライアントは連番付き入力を一定周期で送る。
- 自キャラクターは予測とサーバー照合、他プレイヤーは補間を基本とする。
- 協力ギミック、ゴール、救助はサーバーイベントを正とする。
- 役割、移動速度、ジャンプ力、救助距離、救助待ち時間はサーバー状態を正とする。クライアントの役割能力は操作感を良くする予測であり、snapshotで必ず照合する。
- 許容遅延、tick rate、snapshot rateは実機・実網の計測で決定する。

## 7. ステージとゲームデータ

- 座標や足場をコードへ直書きせず、JSONまたは型付きデータとして管理する。
- ステージID、サイズ、開始地点、チェックポイント、足場、ギミック、ゴールを定義する。
- データ検証をビルド時とサーバー起動時に実行する。
- クライアント描画とサーバー判定で同じ識別子・数値定義を共有する。

## 8. エラー・再接続

- CDNやアセット読込失敗時は、理由と再試行UIを表示する。
- WebSocket切断時は操作を一時保留し、段階的バックオフで再接続する。
- 再接続に失敗した場合、タイトルへ戻す前に状態を明示する。
- サーバーエラーは内部詳細を隠した安定したエラーコードで返す。

## 9. セキュリティと不正対策

- 入力頻度、値範囲、移動量、ルーム操作をサーバーで検証する。
- ルーム参加用トークンと再接続トークンを分離し、有効期限を設ける。
- ログへトークンや個人情報を出力しない。
- 依存関係を固定し、定期的な脆弱性確認を行う。

## 10. テスト戦略

- 単体: 入力変換、移動規則、プロトコル検証、ルーム状態遷移
- 結合: 複数クライアントの参加、開始、同期、切断、再接続、クリア
- E2E: モバイルviewportでの作成→参加→プレイ→結果→再挑戦
- 性能: 6人×複数ルーム、遅延・揺らぎ・パケット欠落の条件
- 手動: iOS Safari、Android Chromeのタッチ、回転、バックグラウンド復帰

## 11. CI/CD

Pull Requestで次を必須化する。

1. 依存関係の再現可能なインストール
2. Format確認
3. ESLint
4. TypeScript型検査
5. 単体・結合テスト
6. クライアント・サーバービルド
7. 主要E2Eスモークテスト

本番配布は、プレビュー環境で同一成果物を検証後、承認付きで昇格する。

## 12. 観測性

- 接続数、アクティブルーム数、tick遅延、メッセージ量
- ルーム参加失敗率、切断率、再接続成功率
- クライアント例外、ロード時間、フレーム時間
- セッション開始、完走、リトライ、協力イベント

個人を不必要に識別せず、計測目的と保持期間を明確にする。

## 13. 移行戦略

1. 既存 `index.html` を基準実装として残す。
2. ワークスペースとクライアント雛形を追加する。
3. 定数、入力、ステージ、プレイヤー、シーンの順に移す。
4. 機能同等性とモバイル動作を確認する。
5. 新クライアントを既定化してから旧実装とZIPを整理する。
6. 共有プロトコル、サーバー、2人縦切りを追加する。
