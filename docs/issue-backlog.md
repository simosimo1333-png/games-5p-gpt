# Issueバックログ

GitHub Issueへ登録する開発タスクの原案。上から順番にすべてを直列実行するのではなく、依存関係を満たしたものを並行して進める。各Issueは実装前に担当者、見積もり、対象マイルストーンを設定する。

## 1. [Foundation] ライセンスと貢献方針を決定する

**優先度:** P0 / **依存:** なし

### 作業

- 公開方針、著作権者、外部素材の条件を確認する
- 採用ライセンスを決定し `LICENSE` を追加する
- `CONTRIBUTING.md` にIssue・PR・レビューの流れを記載する

### 完了条件

- [ ] READMEからライセンスと貢献方法を確認できる
- [ ] 外部素材の利用条件を追跡できる

## 2. [Foundation] ZIP成果物とソースの正本を整理する

**優先度:** P0 / **依存:** なし

### 作業

- `houkago-dash-starter.zip` の内容、生成元、用途を確認する
- リポジトリへ保持するか、Release成果物へ移すか決定する
- 保持する場合は再生成方法とバージョンを文書化する

### 完了条件

- [ ] 編集対象となるソースの正本が一意である
- [ ] バイナリ成果物の更新方法が明確である

## 3. [Game Design] 6人協力プレイのMVP仕様を確定する

**優先度:** P0 / **依存:** なし

### 作業

- 対応人数、役割重複、開始・クリア・失敗条件を決める
- 離脱、途中参加、再接続、人数不足時の扱いを決める
- MVPで実装する協力ギミックを1種類選ぶ
- 決定を `docs/game-design.md` へ反映する

### 完了条件

- [ ] 2人縦切りと6人完成形の受け入れ条件が明文化されている
- [ ] 未決定事項に担当者と決定期限がある

## 4. [Tooling] Vite・TypeScript・npm workspacesを導入する

**優先度:** P0 / **依存:** #2

### 作業

- `apps/client` と共通設定を作成する
- TypeScript strict、開発サーバー、production buildを設定する
- 既存 `index.html` は比較用として維持する

### 完了条件

- [ ] 新クライアントがローカルで起動・ビルドできる
- [ ] 導入手順がREADMEにある
- [ ] 既存ゲームの基準実装が失われていない

## 5. [Quality] Lint・Format・テスト・GitHub Actionsを導入する

**優先度:** P0 / **依存:** #4

### 作業

- ESLint、Prettier、Vitest、Playwrightを設定する
- 型検査、Lint、単体テスト、ビルド、E2EスモークをCI化する
- Pull Requestで必須とするチェック名を文書化する

### 完了条件

- [ ] クリーン環境で全チェックが再現可能に通る
- [ ] 意図的な型・Lint・テスト失敗をCIが検出する

## 6. [Refactor] ゲーム設定とステージデータを分離する

**優先度:** P1 / **依存:** #4

### 作業

- 解像度、重力、速度、ジャンプ力を設定へ移す
- 足場、開始地点、ゴール、チェックポイントを型付きデータへ移す
- ステージデータのバリデーションを追加する

### 完了条件

- [ ] マジックナンバーが責務別の設定へ集約されている
- [ ] コード変更なしでステージデータを差し替えられる

## 7. [Refactor] シーン・プレイヤー・UIの責務を分離する

**優先度:** P1 / **依存:** #4, #6

### 作業

- Boot、Game、Resultシーンへ分割する
- PlayerController、HUD、ゴール処理を分離する
- 現行の移動、落下復帰、ゴール、リトライを維持する

### 完了条件

- [ ] 単一クラスが入力、物理、UI、進行を同時に所有していない
- [ ] 機能同等性の自動または手動確認項目が通る

## 8. [Mobile] 入力・画面回転・セーフエリアを堅牢化する

**優先度:** P1 / **依存:** #7

### 作業

- 入力を共通アクションへ変換するInputControllerを作る
- `pointercancel`、同時タッチ、フォーカス喪失時に入力を解除する
- ノッチ、画面回転、極端なアスペクト比を扱う
- iOS SafariとAndroid Chromeで実機確認する

### 完了条件

- [ ] OS割り込み後に移動入力が残らない
- [ ] 対象実機で操作UIが安全領域内に表示される

## 9. [Protocol] クライアント・サーバー共有プロトコルを定義する

**優先度:** P0 / **依存:** #3, #4

### 作業

- `packages/protocol` を作成する
- ルーム、入力、snapshot、イベント、エラーの型とスキーマを定義する
- プロトコルバージョンと互換性エラーを定義する

### 完了条件

- [ ] 不正メッセージをランタイム検証で拒否できる
- [ ] クライアントとサーバーが同じ定義を利用する

## 10. [Server] 権威サーバーとルーム状態遷移を実装する

**優先度:** P0 / **依存:** #3, #9

### 作業

- WebSocketサーバーと固定tickループを作る
- 作成、参加、準備、開始、終了、退出を実装する
- 人数上限、入力頻度、値範囲をサーバーで検証する

### 完了条件

- [ ] ルーム状態遷移の単体・結合テストがある
- [ ] 7人目、重複参加、不正遷移を安全に拒否する

## 11. [Online] 合言葉によるルーム作成・参加UIを実装する

**優先度:** P1 / **依存:** #7, #9, #10

### 作業

- LobbySceneを追加する
- 作成、参加、準備、エラー、接続状態を表示する
- 招待リンク対応を後から追加できる境界を用意する

### 完了条件

- [ ] 2台の端末が同じルームへ参加して開始できる
- [ ] 満員、無効コード、通信失敗を利用者へ説明できる

## 12. [Online] 2人の移動・ジャンプ同期を実装する

**優先度:** P0 / **依存:** #8, #10, #11

### 作業

- 連番付き入力、サーバーsnapshot、補間を実装する
- 自プレイヤーの予測とサーバー照合を段階的に導入する
- 遅延・揺らぎ・パケット欠落を与えるテストを追加する

### 完了条件

- [ ] 異なる2端末で互いの移動とジャンプを確認できる
- [ ] 通常のモバイル回線条件で大きなワープが継続しない

## 13. [Online] 切断検知と再接続を実装する

**優先度:** P0 / **依存:** #10, #11, #12

### 作業

- 短命な再接続トークンと猶予時間を実装する
- バックグラウンド遷移、回線切替、サーバー切断を扱う
- 再接続中・失敗時のUIを追加する

### 完了条件

- [ ] 一時切断後に同じプレイヤーとして復帰できる
- [ ] 期限切れ・不正トークンを拒否できる
- [ ] 切断・復帰の結合テストがある

## 14. [Gameplay] 最初の協力ギミックと同期クリアを実装する

**優先度:** P1 / **依存:** #3, #12

### 作業

- 2人の協力が必要なギミックを1種類実装する
- 判定、救助、ゴール、タイマー、リトライをサーバー権威にする
- 単独でも進行不能になり続けない救済を設計する

### 完了条件

- [ ] 2人で協力してステージを完走できる
- [ ] 両端末で開始・クリア・結果・再挑戦が一致する

## 15. [Scale] 3〜6人の負荷・同期・UIを検証する

**優先度:** P1 / **依存:** #12, #13, #14

### 作業

- 6人参加時の帯域、tick時間、描画負荷を計測する
- 参加者表示、識別色、画面外プレイヤー案内を追加する
- 複数ルームの負荷試験を追加する

### 完了条件

- [ ] 6人実機テストを繰り返し完走できる
- [ ] 性能予算と測定結果が文書化されている

## 16. [Operations] ログ・メトリクス・障害対応の基盤を追加する

**優先度:** P1 / **依存:** #10

### 作業

- 構造化ログと相関IDを追加する
- 接続、ルーム、tick遅延、切断、再接続、例外を計測する
- トークンや個人情報をログへ残さないルールを設ける
- アラートと最小の障害対応手順を作る

### 完了条件

- [ ] 1セッションの開始から終了をログで追跡できる
- [ ] 主要エラー率と性能劣化を検知できる

## 推奨ラベル

- 種別: `type:feature`, `type:refactor`, `type:docs`, `type:chore`
- 領域: `area:client`, `area:server`, `area:protocol`, `area:game-design`, `area:ops`
- 優先度: `priority:P0`, `priority:P1`, `priority:P2`
- 状態: `status:blocked`, `status:needs-decision`, `good first issue`
