# ソースと成果物の管理方針

## 正式なソース

`main` ブランチで通常のファイルとして確認できる内容を、このプロジェクトの正式なソースとする。ZIPなどのアーカイブ内にだけ存在するコードは、実装済み機能や開発対象として扱わない。

この方針により、すべての変更をGitの差分、Pull Request、レビュー、自動チェックで追跡できるようにする。

## `houkago-dash-starter.zip` の調査結果

ZIPには次のスターター構成が含まれていた。

- Phaser + Vite + TypeScriptのWebクライアント
- Cloudflare Workers + Durable Objectsのリアルタイムサーバー雛形
- 共有プロトコルパッケージ
- アーキテクチャと開発タスクの文書

内容は将来構成の参考になるが、次の理由からリポジトリの正本としては採用しない。

- ZIP内のファイルはGitHub上で直接閲覧・検索・レビューしにくい
- ルートの `index.html` と重複する1人用実装があり、どちらが最新か判断できない
- lockfileがなく、依存関係の完全な再現性がない
- サーバー雛形は入力値検証、認証、再接続、権威シミュレーションが未完成である
- ZIPを更新する手順、生成元コミット、バージョンが記録されていない

ZIPはリポジトリから削除する。必要な場合は、追加時のコミット `f71c0bf` をGit履歴から参照できる。

## 開発中のルール

- ソースコード、設定、文書はアーカイブせず、通常のファイルとしてコミットする。
- ZIP、ビルド出力、依存関係フォルダをソースの代わりにコミットしない。
- スターターZIPのコードを利用する場合は、関連Issueの設計に合わせて内容を見直し、通常のPull Requestとして追加する。
- 生成物には、生成元コミットと再生成コマンドが必要である。
- 秘密情報、環境変数、認証情報を成果物へ含めない。

## 配布物の扱い

配布用ZIPが必要になった場合は、GitHub Releaseまたは配布サービスへ置き、Gitリポジトリには含めない。次の条件が整ってから導入する。

1. 再現可能なビルドコマンドがある
2. CIが同じ成果物を生成できる
3. バージョンと生成元コミットを記録できる
4. 自動テストと必要なセキュリティ確認が完了している

現在のプロトタイプは静的ファイルを直接配信するため、配布用ZIPは不要である。
